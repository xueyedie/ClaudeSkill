# 索引体检与自动化

## 默认做法：直接读取 Notion 数据源

从现在开始，`Knowledge Index` 的日常体检应优先使用 Notion MCP 直接读取与检查，不再把“手工导出 CSV”作为默认主流程。

适用场景：

- 每周例行巡检
- 重整某个主题前的预检查
- 删除文章前的结构一致性检查
- 用户要求“帮我看一下索引有没有问题”

### 推荐巡检顺序

1. 读取 `Knowledge Index 使用说明`
2. 读取 `Knowledge Hub`
3. 读取 `Knowledge Index` 数据库和视图配置
4. 优先查看以下视图：
   - `Default view`
   - `AI`
   - `OpenClaw`
   - `游戏`
   - `Manual Review`
5. 通过搜索与定向读取，检查这些问题：
   - `Domain` 缺失或异常
   - `Parent Topic` 缺失或不合理
   - 管理层对象误入索引
   - 明显重复标题
   - 明显重复 URL 或指向同一原页面的重复索引
   - `Review Cycle` 与 `Last Reviewed` 不匹配
   - canonical 页面缺失或主题下主从关系不清楚

### 直接巡检时的上下文策略

- 不要默认全库逐页读取正文
- 先看索引层，再决定是否需要读正文
- 先看异常项，再决定是否深入
- 只在判断不了时才展开相关页面内容

## 当前工具限制

当前 skill 已经支持“直接用 Notion MCP 做巡检”的工作流，但是否能做到真正意义上的“全表枚举”，取决于当前对话里是否具备可用的数据源查询工具。

因此应采用以下策略：

- 如果当前工具链能直接查询数据源记录，则直接全量巡检
- 如果当前工具链只能做搜索和定向 fetch，则做“索引优先、异常优先”的半自动巡检
- 只有在用户明确需要批量离线扫描、或当前工具链无法高效覆盖时，才退回 CSV 脚本方案

## CSV 脚本的定位

`scripts/index_hygiene.py` 现在改为 **补充方案**，不再是默认方案。

适合这些情况：

- 用户想离线批量扫一遍 CSV 导出
- 当前环境没有足够的 Notion 数据查询能力
- 想做一次独立于 MCP 的快速静态检查

## 每周巡检建议输出

每周巡检结束后，建议直接按报告模板输出，模板见：

- [../assets/weekly-hygiene-report-template.md](../assets/weekly-hygiene-report-template.md)

AI 生成报告时，应尽量包含：

- 本次巡检范围
- 总体健康结论
- 高优先级问题
- 建议修复动作
- 下周前建议处理的事项
